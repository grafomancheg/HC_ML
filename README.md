# HC_ML — Персональные предложения для сети магазинов одежды

Персонализация офферов на базе табличных данных о покупках. Полный контекст
проекта, задачи и подводные камни — в [`CLAUDE.md`](./CLAUDE.md).

Текущий приоритет — модель **«что предложить»** (category propensity):
для каждого клиента считаем вероятность покупки каждой категории в ближайшие
90 дней и берём топ-N в оффер.

## Файлы

| файл | что делает |
|---|---|
| `generate_synthetic_receipts.py` | Пишет правдоподобный `receipts.csv` (2 года, редкие покупки, сезонность, скидочники) для отладки пайплайна без реальных данных. |
| `features.py` | Стартовая четвёрка + дополнительные фичи, все считаются строго до среза `T` (антиутечка). |
| `propensity_model.py` | Скелет пайплайна: срезы `T` по месяцам → длинная таблица `клиент × категория × T` → LightAutoML (fallback LightGBM) → `offers.csv` + метрики + важность фичей. |

## Быстрый прогон end-to-end (на синтетике)

```bash
# 1. Синтетические чеки (~3k клиентов, ~40k строк)
python generate_synthetic_receipts.py --clients 3000 --out receipts.csv

# 2. Модель "что предложить"
python propensity_model.py \
  --receipts receipts.csv \
  --train-start 2024-06-01 --train-end 2025-08-01 \
  --valid-cut 2025-09-01 --score-at 2025-12-01 \
  --horizon-days 90 --top-n 3 \
  --out-offers offers.csv --out-metrics metrics.json \
  --out-importance feature_importance.csv
```

Результат: `offers.csv` (по 3 категории на клиента + score), `metrics.json`
(`precision_at_{1,2,3}`, `recall_at_3`), `feature_importance.csv`.

По умолчанию модель — LightAutoML `TabularAutoML`. Если LAMA не установлен —
прозрачный fallback на `LightGBM` с тем же интерфейсом. Принудительно отключить
LAMA: `--no-lama`.
