# CLAUDE.md — Персональные предложения для сети магазинов одежды

## Контекст проекта

Сеть розничных магазинов одежды + интернет-магазин, есть система лояльности с историей покупок. **Ключевая особенность: покупки редкие — в среднем раз в 3 месяца.** Данные: чеки за 2 года. Цель — рассылать клиентам персональные предложения (что купить), используя AutoML по табличным данным.

Стек: Python, **LightAutoML (LAMA)**, Docker, cron, таймзона Europe/Moscow. Развёртывание — в том же контуре, что и существующие боты.

## Набор задач и приоритет

Идентифицировано несколько ML-задач. Делаем **строго по очереди**, не всё сразу:

1. **«Что предложить» (category propensity) — ТЕКУЩИЙ ПРИОРИТЕТ.** Для каждого клиента — вероятность покупки каждой категории в ближайшие 90 дней. Берём топ-2-3 в оффер.
2. «Когда слать» (time-to-next-purchase) — следующий этап. Предсказывает день готовности к покупке.
3. «Нужна ли скидка» (uplift) — позже. Требует данных о прошлых рассылках (кто получил оффер, кто нет).

Модели работают в паре: №1 даёт *что*, №2 даёт *когда*, №3 — *кому скидка*.

## Модель «что предложить» — формулировка

- **Задача:** бинарная классификация. Не мультикласс «одна следующая категория», а отдельный скор на каждую пару `клиент × категория`.
- **Формат таблицы — «длинный»:** одна строка = `клиент × категория × дата_среза`. Категория — это признак.
- **Таргет:** купил ли клиент эту категорию в окне `[T, T+90 дней]` → 1/0.
- **Антиутечка (КРИТИЧНО):** все признаки считаются строго по истории **до** даты среза `T`. Окно таргета модели при обучении не показывается.
- **Трюк против разреженности:** делаем много срезов `T` (например, на начало каждого месяца за 2 года). Один клиент → десятки обучающих строк. Так из редких покупок получается богатый обучающий набор.
- **Валидация:** временной сплит (последний срез — в валидацию), без перемешивания. Метрика качества — **Precision@N** (из топ-N предложенных категорий сколько реально куплено), НЕ ROC-AUC (он обманчиво высок из-за дисбаланса).
- Стартовый код — в файле `propensity_model.py` (скелет на LightAutoML: построение срезов, фичи, обучение, генерация `offers.csv`, важность фичей).

## Данные

Нужна одна хорошая выгрузка → файл `receipts.csv` (одна строка = одна позиция в чеке):

| поле | описание |
|---|---|
| `client_id` | ID клиента (карта лояльности) — **единый ключ** |
| `date` | дата/время покупки |
| `category` | категория товара (через справочник из артикула) |
| `brand` | бренд |
| `amount` | сумма по позиции |
| `discount` | была ли скидка / размер скидки |
| `channel` | магазин или сайт |
| `size`, `color` | если касса пишет |

Отдельно — карточки клиентов (`client_id`, имя, e-mail/телефон, **согласие на рассылку — 152-ФЗ**, пол, город, дата регистрации) и справочник товаров (артикул → категория, бренд, пол, сезон, цена).

**Главная точка отказа:** `client_id` должен быть **одинаковым** во всех выгрузках, иначе покупки не сшить с контактами. Проверять в первую очередь.

## Признаки (фичи)

Большинство считается простым счётом по чекам — работают и как фичи для модели, и как самостоятельные правила рассылки без ML.

**Стартовая четвёрка** (максимум пользы, минимум усилий, без ML):
- **RFM-сегмент** — давность / частота / сумма → сегменты (лучшие, новички, спящие, уходящие).
- **Любимый бренд** — бренд с макс. долей трат (устойчиво от 2–3 чеков).
- **Любимый размер** — самый частый размер (выше отклик, меньше возвратов).
- **Скидочник или нет** — доля покупок по акции (зерно uplift-логики: не давать скидку тем, кто и так купит).

**Дополнительно:** ценовой уровень, любимые цвета, категории и их связки, сезонность клиента, ритм покупок (интервал → для модели «когда»), любимый канал, признак оттока, доля возвратов, пол / для кого покупает.

Полная шпаргалка «признак → как считается → как использовать» — в отдельном документе проекта.

## Пайплайн доставки (от таблицы к клиенту)

1. **Правила и фильтры** — здравый смысл, которого модель не знает: нет в наличии — не предлагать; купил вчера — не предлагать; нет согласия — не слать; учесть размер.
2. **Канал** — персональный оффер ≠ пост в канале. Нужен 1-to-1: e-mail / SMS / пуш / личное сообщение. Контакты — из системы лояльности.
3. **Личное сообщение** — шаблон с «дырками» `{имя}`, `{категория}`, `{бренд}`, картинка/лук, ссылка. Один шаблон → тысячи персональных писем. Сюда встраивается бот-коллаж (готовый лук вместо слова «платья»).
4. **Отправка** — ночной cron: пересчёт таблицы → формирование сообщений → отдача в рассылочный сервис / систему лояльности. Docker-контур.
5. **Замер** — контрольная группа (часть клиентов офферов НЕ получает) → сравнить покупки/выручку между группами. Единственное честное доказательство в деньгах.
6. **Обратная связь** — кто купил после оффера, кто нет → новые данные → обратно в ночной пересчёт. Модель улучшается с каждым циклом.

## Дорожная карта

1. Получить выгрузку по **одному магазину/сезону**, проверить поля и единый `client_id`.
2. Собрать `receipts.csv`, посчитать стартовую четвёрку признаков.
3. Запустить **простую персонализацию по правилам** (RFM + бренд + размер + новинки любимого бренда) — уже без ML.
4. Прогнать модель «что предложить» (`propensity_model.py`), усилив её этими признаками.
5. Настроить контрольную группу и замер (Precision@N + A/B).
6. Масштабировать на всю сеть.
7. Позже: модель «когда слать», затем uplift.

## Ограничения и подводные камни

- **Утечка из будущего** — главный риск. Строго следить за датами срезов.
- **Новые клиенты (1–2 покупки)** — фичей почти нет, персонализация слабая. Вести через общие новинки/популярное.
- **Сезонность одежды** — учитывать месяц/сезон среза; валидировать на сопоставимых сезонах.
- **Частота рассылок** — покупки редкие, не заваливать: 1–2 уместных оффера в месяц максимум, иначе отписки.
- **152-ФЗ** — без согласия не писать, каким бы точным ни был прогноз.
- **Начинать с малого** — один магазин, один сегмент, один замер. Убедились — расширяем.
